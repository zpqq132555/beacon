当前 Beacon 已完成运行时命名私有化、平台范围收敛和私有供应链第一阶段落地，但 `beacon init` 仍保留几处明显不符合私有版目标的历史包袱：
- init 入口 banner 仍带有旧时期视觉残留，和 Beacon 品牌不完全一致。
- init / update 仍暴露语言选择与双语技能分发模型，而当前私有化方向已经明确后续只维护中文运行时资产。
- init 只负责安装平台技能与可选依赖，还没有像 Claude Code `/init` 那样，为接入项目初始化 AI 可直接消费的协作上下文文件。

这次讨论的目标不是继续推进项目级私有供应链 rollout，而是把 init 收敛成 Beacon 私有版真正的项目入口。

### Q1：这次 change 是继续做项目接入流程，还是转向工作流入口优化？

结论：转向工作流入口优化，并以 `init` 作为第一优先级。

原因：
- 项目级接入、升级、回滚合同已经在上一轮 change 中补齐，继续深挖的边际收益变低。
- `init` 是用户第一次接触 Beacon 私有版的入口，品牌、分发模型和上下文初始化都在这里汇合。
- 后续无论工作流本身怎么演进，`init` 的私有化收口都能直接提升首轮接入体验。

### Q2：`init` 这次要解决哪些问题？

候选方向：

方案 A：只修 banner 和文案残留
- 优点：范围最小，回归风险最低。
- 缺点：只能解决品牌表层问题，语言分发和上下文初始化两个更核心的问题仍然存在。

方案 B：修 banner + 去语言化
- 优点：能明显收敛私有版运行时边界，减少 `init` / `update` / 资产分发测试中的双语复杂度。
- 缺点：仍然缺少项目级协作上下文初始化能力，`init` 只是安装器，不是真正的项目入口。

方案 C：修 banner + 去语言化 + Agent 上下文初始化
- 优点：一次把品牌、分发模型和项目入口能力收口完整；`init` 安装完成后，目标仓库会立即具备 AI 可执行的维护上下文。
- 缺点：范围最大，需要同时改 CLI、技能资产分发、文档和测试，并新增目标仓库文档生成逻辑。

推荐：方案 C。

### Q3：语言模型要怎么收口？

讨论结果：
- 私有版后续不再维护英文运行时技能，不再继续扩展双语运行时分发。
- `beacon init` 不再提示技能语言选择，也不接受语言多选语义。
- `beacon update` 也应与 init 保持同一分发合同，改为中文单轨刷新，而不是继续检测或切换技能语言。
- `assets/skills-zh/` 将成为运行时唯一分发来源；英文资产若保留，也只作为历史或迁移期参考，不再参与用户可见分发合同。

### Q4：Agent 上下文初始化要不要一起做？

结论：要，而且只在 `scope=project` 下启用。

原因：
- 这是项目级入口能力，不适合在 `global init` 中去扫描和写项目文档。
- 私有版 Beacon 的目标不是只把技能装进去，还要让目标仓库立刻具备 AI 可执行的协作上下文。
- 和品牌、语言分发一起做，可以一次定义清楚 init 结束时目标仓库应该处于什么状态。

### Q5：生成的上下文文件是给谁看的？

结论：优先给 AI 看，不是给人看的 onboarding 文档。

因此根文件必须满足：
- 短
- 具体
- 高信号
- 可执行
- 少背景说明，多约束和命令

根 `AGENTS.md` 只保留：
1. Stack
2. Commands
3. Rules
4. Verify
5. Structure
6. Local AGENTS

目标长度控制在大约 30-60 行，不写成长篇 README 替身。

### Q6：`CLAUDE.md` 怎么处理？

结论：`AGENTS.md` 是唯一维护源，`CLAUDE.md` 只保留 Claude Code 兼容 shim，固定写法为：

```md
@AGENTS.md
```

不复制规则正文，不再双处维护。

### Q7：已有 `AGENTS.md` 时怎么做？

结论：默认不覆盖。

交互模式建议提供三种维护路径：
- 保留现有，仅输出建议
- 基于现有内容合并更新
- 先查看建议，再决定是否改写

非交互 `--yes` 模式：
- 若不存在 `AGENTS.md`，生成根文件并生成 `CLAUDE.md` shim
- 若已存在 `AGENTS.md`，默认跳过正文写入，只补 `CLAUDE.md` 缺失项并输出维护建议

### Q8：局部 `AGENTS.md` 要不要自动批量生成？

结论：先不默认批量生成，只输出建议。

原因：
- 自动在多个目录铺文件容易产生噪音和后续维护负担。
- 更稳的做法是：根 `AGENTS.md` 保持精简，扫描后只提示哪些目录适合补局部规则。

### Q9：要支持哪些项目模板？

结论：首期生成模板覆盖三类常见目标项目：
- Node/TypeScript
- Unity
- Cocos

每类模板都遵循“AI 导向极简骨架”，通过项目扫描填入命令、目录和规则，不生成大段背景说明。

### Agreed Approach

创建一个围绕 `beacon init` 的私有化收口 change，范围包括：
- 修正 init banner 与相关 CLI 品牌残留，确保 Beacon-only 身份完整。
- 移除语言选择与双语运行时分发模型，收敛为中文单轨分发，并同步 update/doc/test 合同。
- 在项目级 init 完成后，扫描目标项目并生成 AI 导向的最小 `AGENTS.md`，同时生成 `@AGENTS.md` 形式的 `CLAUDE.md` shim。
- 对已有 `AGENTS.md` 采用保守维护策略：默认不覆盖，只给建议或经确认后合并更新。
- 对大型仓库只输出局部 `AGENTS.md` 建议，不默认批量生成子目录文件。
