当前 Beacon 已完成 CLI init 去多语言收口，但用户进一步澄清后确认：`AGENTS.md / CLAUDE.md` 的维护不应该继续放在 CLI `init` 里，而应该转成一个独立的 AI 技能来处理。

这次讨论的核心不是继续推进 CLI 接入，而是定义一个项目级 AGENTS 树维护器。

### Q1：这个能力应该挂在哪里？

候选方案：

方案 A：继续塞进 CLI `beacon init`
- 优点：用户入口少
- 缺点：CLI 阶段并没有 AI 介入，职责边界错误，之前已经因此把模板和行为做偏

方案 B：挂在 `/beacon` 主 skill 里
- 优点：主入口统一
- 缺点：`/beacon` 本质是五阶段工作流调度器，继续塞项目上下文维护会让主流程越来越重

方案 C：新增独立 skill `/beacon-init`
- 优点：职责清晰，可直接调用，也可被 archive 阶段转调复用
- 缺点：多一个 skill 入口

结论：选择方案 C，新建独立 skill `/beacon-init`。

### Q2：`/beacon-init` 的职责是什么？

结论：
- `/beacon-init` 是项目 AGENTS 树维护器
- 负责根 `AGENTS.md`
- 负责子目录 `AGENTS.md`
- 负责职责文档 `[职责].md`
- 根目录 `CLAUDE.md` 始终只是以根 `AGENTS.md` 为准的 shim

### Q3：手动调用和 archive 调用是否同一种模式？

候选方案：

方案 A：统一全量维护
- 优点：实现简单
- 缺点：archive 阶段太重，也容易在不值得沉淀时打断用户

方案 B：手动全量维护，archive 增量沉淀
- 优点：符合两种触发场景的语义差异
- 缺点：内部需要区分两套决策路径

结论：选择方案 B。

### Q4：手动调用 `/beacon-init` 时怎么工作？

结论：
- 这是“全量维护”模式
- 如果没有 AGENTS 体系，则创建
- 如果已有 AGENTS 体系，则基于当前项目内容做整体更新
- AI 可以主动判断并创建/更新
- 依据是当前工作区现状，包含未提交改动

### Q5：archive 阶段调用 `/beacon-init` 时怎么工作？

结论：
- 这是“归档增量沉淀”模式
- 如果判断本次归档内容不适合写入 AGENTS，则完全忽略，不提示、不打断流程
- 只有判断适合写入时，才给摘要级建议，并等待用户确认
- 确认后再执行创建/更新/删除

细分情况：
1. 不适合沉淀：不调用，不提示
2. 适合沉淀，且项目里完全没有 AGENTS 体系：走“全量维护 + 本次归档内容注入”
3. 适合沉淀，且项目里已有 AGENTS 体系：只做“本次归档相关的增量维护”

### Q6：根 `AGENTS.md` 应该写什么？

结论：根 `AGENTS.md` 必须保持精简，只承担四类信息：
1. 全局入口
2. 关键命令
3. 高价值全局注意事项
4. AGENTS Map

进一步约束：
- 根 `AGENTS.md` 不承载大量局部细则
- 根 `AGENTS.md` 不复制子文档内容
- 子文档也不重复根里的全局规则，除非明确需要局部补充或覆写

### Q7：子层文档应该怎么组织？

讨论后结论不是单一模板，而是“按职责边界分层，不按目录深度分层”。

#### 7.1 单一职责目录

如果目录边界和职责边界一致，则直接使用目录级 `AGENTS.md`，例如：
- `/tools/ci/AGENTS.md`
- `/tools/codegen/AGENTS.md`
- `/tools/release/AGENTS.md`

#### 7.2 混合目录

如果某目录本身是混合区，则该目录先有一个入口型 `AGENTS.md`，用于导航与约束。

在这种情况下，可以额外有多个职责文档 `[职责].md`，承载这个目录层本身的混合职责规则，例如：
- `/tools/AGENTS.md`
- `/tools/ci.md`
- `/tools/codegen.md`
- `/tools/release.md`

#### 7.3 混合目录 + 单一职责子目录并存

允许同时存在：
- `/tools/AGENTS.md`
- `/tools/[职责].md`
- `/tools/*/AGENTS.md`

适用前提：
- `/tools` 这一层自己就有独立于子目录之外的混合职责
- `/tools/*` 各自又已经是单一职责目录

也就是说可以存在：
- `/tools/AGENTS.md`
- `/tools/root-scripts.md`
- `/tools/ci/AGENTS.md`
- `/tools/codegen/AGENTS.md`
- `/tools/release/AGENTS.md`

### Q8：是否允许更深层级的 AGENTS？

结论：
- 允许，但不默认一路往下创建
- 默认优先两层：根 `AGENTS.md` + 子层规则
- 只有当某子区域已经明显过宽，且下一级存在长期稳定、职责清晰、规则明显不同的区域时，才继续下钻

核心原则：
- AGENTS 分层按职责清晰度建，不按目录深度建

### Q9：创建子规则时，是优先子目录 `AGENTS.md` 还是 `[职责].md`？

结论：
- 目录边界清晰时，优先 `子目录/AGENTS.md`
- 目录边界混合时，使用 `本地 AGENTS.md + 多个 [职责].md`
- `[职责].md` 不能替代本地入口型 `AGENTS.md`

### Q10：是否允许对已有 AGENTS 树做结构优化？

结论：允许。

但按场景分开：
- 手动 `/beacon-init`：允许主动做结构优化，包括合并、收缩、迁移、删除冗余节点
- archive 触发：即使要做结构调整，也必须先给摘要建议，用户确认后再执行

### Q11：旧 AGENTS 文档怎么处理？

最初考虑过保留旧文件并标记 deprecated，但用户明确否定了这种做法。

结论：
- 不保留废弃 AGENTS 文档在工作树中占上下文
- 如果旧 `AGENTS.md` / `[职责].md` 已被新结构替代，就进入建议删除集合
- 用户确认后直接删除
- 历史回溯交给 `git` / `SVN`

### Q12：删除旧文档时是自动删还是建议后删？

结论：
- 即便是手动 `/beacon-init`，也先给出创建 / 更新 / 合并 / 删除的摘要建议
- 用户确认后，再执行删除
- 这样能避免误删，也避免中途消耗上下文去保留过时文件

### Q13：归档时给用户看的建议粒度是什么？

结论：
- 先给摘要级建议，不直接抛出完整 diff
- 摘要只说明：
  - 建议影响哪些 AGENTS 文件
  - 是新增、更新、合并还是删除
  - 为什么这次内容值得沉淀
- 用户确认后再执行具体写入

### Q14：什么样的归档内容值得沉淀到 AGENTS？

结论：总体偏保守，只沉淀长期有效、对 AI 高价值的规则，例如：
- 稳定的目录职责与边界
- 稳定的开发/验证命令与流程约束
- 反复出现、对 AI 很关键的实现规则或禁区

不适合沉淀的包括：
- 一次性需求背景
- 临时 workaround
- 只服务某个单次 feature 的说明

### Agreed Approach

创建一个新的 OpenSpec change，为 `/beacon-init` 建立独立能力，范围包括：
- 新增独立 skill `/beacon-init`
- 定义手动全量维护与 archive 增量沉淀两种模式
- 定义根 `AGENTS.md` 的精简职责
- 定义目录级 `AGENTS.md`、职责文档 `[职责].md` 和更深层级规则的准入边界
- 定义创建 / 更新 / 合并 / 删除的建议与确认机制
- 定义 archive 阶段何时静默忽略、何时给出摘要建议并转调 `/beacon-init`
