## Context

Beacon 已明确不再在 CLI `init` 中维护 `AGENTS.md / CLAUDE.md`，但项目协作仍然需要一个稳定的 AGENTS 树维护入口。用户希望这个入口既能在新项目或老项目中手动全量整理 AGENTS，也能在 archive 阶段按需把本次 change 中稳定、高价值的规则沉淀到 AGENTS，而不是继续靠人工长期维护。与此同时，根 `AGENTS.md` 必须保持精简，局部规则需要按职责边界渐进下沉。

## Goals / Non-Goals

**Goals:**
- 新增独立 skill `/beacon-init`，承担项目级 AGENTS 树维护职责。
- 区分手动全量维护与 archive 增量沉淀两种模式。
- 固定根 `AGENTS.md` 的精简职责，避免继续膨胀成 README 替身。
- 为目录级 `AGENTS.md`、职责文档 `[职责].md` 与更深层级节点建立统一准入规则。
- 允许对已有 AGENTS 树做结构优化，并在确认后删除被替代的旧节点。

**Non-Goals:**
- 不把 AGENTS 维护能力重新塞回 CLI `beacon init`。
- 不要求每个目录都必须拥有自己的 `AGENTS.md`。
- 不把一次性需求背景、临时 workaround 或仅单次 feature 有效的说明默认沉淀到 AGENTS。
- 不把版本历史保留在工作树中的废弃 AGENTS 文档里。

## Decisions

### D1：使用独立 `/beacon-init` skill，而不是 CLI 或 `/beacon` 主 skill
- **选择**：新增独立 `/beacon-init`，可直接调用，也可被 archive 阶段转调。
- **理由**：AGENTS 维护是 AI 参与后的项目上下文治理问题，不属于 CLI 安装职责；同时 `/beacon` 主 skill 作为五阶段流程调度器，不应继续吸收额外的项目治理职责。
- **已考虑 alternative**：继续放在 CLI `init` 或挂进 `/beacon` 主 skill。拒绝原因分别是职责边界错误、主流程过重。

### D2：将 `/beacon-init` 分为手动全量维护与 archive 增量沉淀两种模式
- **选择**：手动调用时按当前工作区做全量维护；archive 阶段只对本次归档内容做增量沉淀。
- **理由**：两种场景的上下文来源、风险容忍度和打断成本不同。手动场景适合主动整理整个 AGENTS 树，archive 场景更适合只沉淀本次 change 中已验证稳定的内容。
- **已考虑 alternative**：统一全量维护。拒绝原因是 archive 阶段会过重，也会放大误沉淀风险。

### D3：根 `AGENTS.md` 固定保持精简
- **选择**：根 `AGENTS.md` 只保留全局入口、关键命令、高价值注意事项和 AGENTS Map。
- **理由**：根文档是高频上下文入口，必须短、小、稳，便于 AI 快速理解全局导航，不应承载大量局部细则。
- **已考虑 alternative**：在根文档中直接写入大量目录规则。拒绝原因是容易膨胀、重复和漂移。

### D4：AGENTS 树按职责边界分层，而不是按目录深度分层
- **选择**：目录边界清晰时优先使用 `子目录/AGENTS.md`；目录混合时使用 `本地 AGENTS.md + [职责].md`；只有在职责继续细分且长期稳定时才允许更深层级节点。
- **理由**：实际项目目录结构经常不完美，若机械按深度建树，会产生很多低价值节点。按职责清晰度分层更适合老项目与长期演化项目。
- **已考虑 alternative**：统一只用 `AGENTS.md`，或统一只用 `[职责].md`。拒绝原因分别是混合目录会变胖、入口文件会丢失。

### D5：混合目录允许入口型 `AGENTS.md`、职责文档和单一职责子目录并存
- **选择**：允许例如 `/tools/AGENTS.md`、`/tools/root-scripts.md`、`/tools/ci/AGENTS.md` 同时存在。
- **理由**：有些目录本身是混合区，同时又包含边界清晰的子目录。如果强行二选一，会损失表达能力。
- **已考虑 alternative**：混合目录只能二选一。拒绝原因是不能覆盖复杂项目的真实结构。

### D6：归档阶段只在“值得沉淀”时出现摘要建议
- **选择**：archive 阶段默认静默；只有当 AI 判断本次归档内容适合写入 AGENTS 时，才给出摘要级建议并等待用户确认。
- **理由**：避免每次归档都打断用户，也避免把不稳定信息写入长期上下文。
- **已考虑 alternative**：archive 阶段每次都提示一次。拒绝原因是噪音过大，长期会降低用户接受度。

### D7：手动模式与 archive 模式都采用“先摘要建议，后执行”的删除策略
- **选择**：创建、更新、合并、删除都先出摘要建议；用户确认后才执行删除旧 AGENTS 文档。
- **理由**：即使手动模式允许更主动的治理，也不应在没有确认的情况下删除现有上下文文件。
- **已考虑 alternative**：先标记 deprecated、保留旧文件。拒绝原因是过时文档会继续占用模型上下文，历史追溯应交给版本控制系统。

### D8：手动模式以当前工作区现状为准
- **选择**：手动 `/beacon-init` 读取当前工作区，包括未提交改动，并在工作区脏时提醒用户本次整理基于当前状态。
- **理由**：新项目初始化和老项目接入这套工具时，很多真实职责边界尚未提交；如果只看已提交状态，会漏掉当前最需要沉淀的信息。
- **已考虑 alternative**：只依据已提交状态。拒绝原因是会和用户实际整理诉求脱节。

## Risks / Trade-offs

- [Risk] AGENTS 树分层规则过于灵活，可能导致不同项目之间结构风格漂移。
  → Mitigation: 把根文档职责、目录级入口职责和职责文档准入规则写成明确 contract，并在 skill 中内置优先级。
- [Risk] archive 阶段对“是否值得沉淀”的判断若过宽，可能把一次性规则写进长期上下文。
  → Mitigation: 默认偏保守，只沉淀长期有效、对 AI 高价值的信息，不适合时完全静默忽略。
- [Risk] 删除旧 AGENTS 文档存在误删风险。
  → Mitigation: 先输出摘要建议，明确列出待删除项与替代去向，用户确认后再删除。
- [Trade-off] 不保留 deprecated 文档意味着工作树里没有显式过渡态。
  → 接受理由：版本历史已经由 git / SVN 承担，保持当前上下文树干净更重要。
- [Trade-off] 手动模式读取未提交改动，可能让 AGENTS 描述更贴近临时状态。
  → 接受理由：这是用户显式整理当前项目结构时更需要的视角；通过脏工作区提醒降低误解。

## Migration Plan

1. 为 `/beacon-init` 设计独立 skill 入口、参考文档与分发资产。
2. 定义 AGENTS 树拓扑规则，包括根文档、目录级入口、职责文档和更深层级节点的准入边界。
3. 为 `/beacon-init` 实现手动全量维护与 archive 增量沉淀两套行为合同。
4. 在 archive 流程中加入“静默忽略 / 摘要建议 / 用户确认后转调 `/beacon-init`”的衔接逻辑。
5. 补充创建、更新、合并、删除及静默忽略场景的回归测试与文档说明。

Rollback:
- 若 `/beacon-init` 技能设计未达到稳定状态，可仅保留手动模式，暂不接入 archive 流程。
- 若 archive 阶段的沉淀判断过于激进，可回退到“始终不在 archive 中自动触发，只允许手动调用”的更保守模式。
- 本 change 不依赖数据迁移；删除类行为在确认后才执行，因此回滚主要是行为合同回退。

## Open Questions

- `/beacon-init` 的建议摘要是否需要固定格式，以便未来测试更稳定地断言其内容边界。
- archive 转调 `/beacon-init` 时，是否需要显式把当前 change 的 proposal/design/tasks/spec 路径传入，还是由 skill 自行发现最近归档上下文。
- 对已有但质量较差的 AGENTS 树，是否需要定义“只修内容，不调结构”的保守模式作为后续扩展。
